# UmbraVox Cryptographic Safety Reference

This document records all nonce constructions and HKDF domain-separation
strings used in UmbraVox.  It supersedes the design sketches in
`attic/doc-legacy-2026-04-28/hardening/07-nonce-safety.md`, which described
a prototype nonce layout that was never shipped.

---

## 1. Nonce Constructions

All AEAD nonces in UmbraVox are 12 bytes (96 bits).  Three independent
constructions are used; each is described below together with the code
location that implements it.

### 1.1 Double Ratchet (Signal layer) — HKDF-derived base XOR counter

**Source:** `src/UmbraVox/Crypto/Signal/DoubleRatchet.hs`, function
`makeNonce`.

```
nonce = zeros[4] || ( HKDF(salt=zeros[32], ikm=chainKey,
                           info="UmbraVox_Nonce_v1")[0:8]
                      XOR LE64(counter) )
```

Step-by-step:

1. **PRK** = HKDF-Extract(salt=`0x00`×32, IKM=`chainKey`)
   (zero salt follows Signal/RFC 5869 §2.2 convention so Extract acts as a
   keyed PRF on `chainKey` alone)
2. **base**[0:8] = HKDF-Expand(PRK, info=`"UmbraVox_Nonce_v1"`, L=8)
3. **ctr**[0:8] = `LE64(counter)` (8-byte little-endian encoding of the
   `Word32` message counter, zero-extended to 64 bits)
4. **mixed**[0:8] = base XOR ctr
5. **nonce** = `0x00 0x00 0x00 0x00` || mixed

Properties:

- `chainKey` is always distinct from the GCM `msgKey` (the two are derived
  from different HMAC calls on the same chain key: `msgKey = HMAC(ck, 0x01)`,
  `newChainKey = HMAC(ck, 0x02)`).  Key separation is maintained.
- The XOR with `LE64(counter)` guarantees uniqueness within a chain: each
  message counter value is used at most once per chain key.
- A DH ratchet step replaces `chainKey` entirely, making counter resets to 0
  safe.
- The nonce is passed to AES-256-GCM; the `chainKey` itself is never given
  to GCM directly.

### 1.2 Noise Transport Layer — Noise §11.3 / RFC 8439 counter nonce

**Source:** `src/UmbraVox/Network/Noise/State.hs`, function `makeNonce`.

```
nonce = zeros[4] || LE64(counter)
```

The counter is a per-direction monotonic `Word64` (`nsSendN` / `nsRecvN`).
It starts at 0 and is never reset within a session.  This layout matches the
Noise Protocol Framework specification §11.3 (ChaCha20-Poly1305 nonce
encoding) and RFC 8439 §2.6.

Properties:

- No HKDF derivation step; the counter directly occupies bytes 4–11 in
  little-endian order.
- The send key (`nsSendEncKey`) and receive key (`nsRecvEncKey`) are separate
  32-byte ChaCha20-Poly1305 keys derived during the Noise IK handshake;
  they are never reused across sessions.
- Every transport message additionally binds the 32-byte final handshake hash
  (`nsHandshakeHash`) as AAD, providing channel binding per Noise §11.2.

### 1.3 Storage Field Encryption — random per-field nonce

**Source:** `src/UmbraVox/Storage/Encryption.hs`, function `encryptField`.

```
nonce = randomBytes(12)   -- fresh CSPRNG draw per field write
```

The 12-byte nonce is generated by `UmbraVox.Crypto.Random.randomBytes` for
each call to `encryptField`.  It is stored as the first 12 bytes of the
`UVENC1:` blob:

```
"UVENC1:" || hex( nonce[12] || ciphertext || tag[16] )
```

The storage key is derived per-installation from a random 32-byte salt
(loaded or generated by `getOrCreateSalt`) via HKDF-SHA-256 with info
`"UmbraVox_StorageAEAD_v1"`, so the (key, nonce) birthday bound applies
independently per installation.  AES-256-GCM is used; AAD is empty.

## 2. HKDF Domain Separation Strings

All `info` labels used in HKDF-Expand calls across the codebase.  Every
label must be unique to prevent cross-context key confusion.

| Info string | Module | Usage |
|---|---|---|
| `"UmbraVox_Ratchet_v1"` | `Crypto.Signal.DoubleRatchet` (`ratchetInfo`) | Root-key + chain-key derivation in `kdfRK`; salt = current root key, IKM = X25519 DH output |
| `"UmbraVox_Nonce_v1"` | `Crypto.Signal.DoubleRatchet` (`nonceInfo`) | 8-byte nonce base in `makeNonce`; salt = zeros, IKM = chain key |
| `"UmbraVox_X3DH_v1"` | `Crypto.Signal.X3DH` (`x3dhInfo`) | Master-secret derivation from DH outputs; info is `label || IK_A_pub || IK_B_pub` for identity binding |
| `"UmbraVox_PQXDH_v1"` | `Crypto.Signal.PQXDH` (`pqxdhInfo`) | Post-quantum extended DH master-secret derivation; info is `label || IK_A_pub || IK_B_pub` |
| `"UmbraVox_StealthKey_v1"` | `Crypto.StealthAddress` (`stealthKeyInfo`) | Stealth payment key derivation |
| `"UmbraVox_ViewTag_v2"` | `Crypto.StealthAddress` (`viewTagInfo`) | View-tag derivation for stealth scanning |
| `"UmbraVox_KeyStore_v1"` | `Crypto.KeyStore` (`keystoreInfo`) | Key-store encryption key derivation |
| `"UmbraVox_Export_v1"` | `Crypto.Export` (`exportInfo`) | Key-export / backup derivation |
| `"UmbraVox_StorageAEAD_v1"` | `Storage.Encryption` (inline) | Per-installation storage key derivation; salt = random per-install salt |
| `"UmbraVox_SafetyNumber_v2"` | `Protocol.QRCode` (inline) | Safety-number derivation from identity fingerprints |

The Noise prologue (`"UmbraVox_v1"`) and protocol name
(`"Noise_IK_25519_ChaChaPoly_SHA256"`) are mixed into the Noise handshake
hash `h0` and are not HKDF info strings; they are listed here for
completeness only.

The authoritative registry of info strings is the comment block in
`src/UmbraVox/App/Defaults.hs` (starting at the `-- HKDF domain-separation
strings` heading).  If a new info string is added anywhere in `src/`, it
must also be added to that registry and to the table above.

## 3. Key Hierarchy Summary

```
PQXDH master secret
  └─ HKDF("UmbraVox_PQXDH_v1") ──► session root key
       └─ kdfRK("UmbraVox_Ratchet_v1")
            ├─► new root key
            └─► chain key (send or recv)
                 ├─ HMAC(ck, 0x01) ──► msgKey  (→ AES-256-GCM encryption)
                 ├─ HMAC(ck, 0x02) ──► newChainKey
                 └─ HKDF("UmbraVox_Nonce_v1") ──► nonce base (→ XOR counter → GCM nonce)

X3DH master secret
  └─ HKDF("UmbraVox_X3DH_v1") ──► X3DH shared secret ──► Double Ratchet root key

Noise IK handshake
  └─ (MixKey steps per Noise spec) ──► nsSendEncKey, nsRecvEncKey (ChaCha20-Poly1305)
       └─ makeNonce(counter) ──► zeros[4] || LE64(n)

Identity secret + per-install salt
  └─ HKDF("UmbraVox_StorageAEAD_v1") ──► StorageKey (AES-256-GCM, random nonce)
```

## 4. Limitations and Known Gaps

- **Key erasure in Haskell**: `ByteString` values are immutable and
  GC-managed; there is no guaranteed way to zero key material after use from
  pure Haskell.  The `UmbraVox.Crypto.SecureBytes` module provides a
  best-effort `ForeignPtr`-based container with a zeroing finalizer, but the
  finalizer uses `pokeByteOff` which the compiler may elide.  Production-grade
  erasure requires an FFI call to `explicit_bzero` (Linux) or `SecureZeroMemory`
  (Windows).  This is tracked as a known limitation pending the FFI hardening
  milestone.

- **Counter persistence**: The Double Ratchet counter is held in memory; a
  process crash between persisting state and completing encryption can waste a
  counter value (safe) but correct crash-recovery requires the
  write-before-encrypt discipline described in
  `attic/doc-legacy-2026-04-28/hardening/07-nonce-safety.md` §3.

- **Noise counter exhaustion**: `nsSendN` and `nsRecvN` are `Word64`; at
  2^64 messages per session key the nonce space is exhausted.  Sessions must
  be renegotiated before this limit.  In practice UmbraVox performs session
  rotation long before this bound.

- **Ephemeral mode and peer logging**: Ephemeral mode eliminates YOUR forensic
  trail but does NOT prevent the peer from logging messages.  Running in
  ephemeral mode removes local persistence of identity keys, message history,
  and peer records, but has no effect on what the remote peer stores.  Users
  should not treat ephemeral mode as a guarantee of end-to-end deniability
  unless the peer is also verified to be running without persistent storage.
