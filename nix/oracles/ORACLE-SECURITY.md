# Oracle Supply Chain Security

## Purpose

Before trusting any external library as a reference oracle for differential
testing, we review its security posture. This file documents the review.

## Oracle Trust Hierarchy

1. **F* executable specs** — highest trust (our verified formal model)
2. **HACL\*/EverCrypt** — formally verified in F*, extracted to C
3. **Official test vectors** — RFC/NIST/Wycheproof (standards bodies)
4. **Audited implementations** — libsodium, Monocypher, libsignal
5. **Widely-deployed implementations** — BoringSSL, RustCrypto
6. **Research implementations** — PQClean, liboqs

## Per-Oracle Review

### HACL\* / EverCrypt
- **Provenance**: Microsoft Research + INRIA
- **Verification**: Formally verified in F*, machine-checked extraction via KaRaMeL
- **Audit**: The verification IS the audit — proofs cover memory safety, functional
  correctness, and secret independence (constant-time)
- **Known issues**: None affecting our tested primitives
- **Independence**: Completely independent codebase from all other oracles

### libsodium
- **Provenance**: Frank Denis (jedisct1), open source since 2013
- **Audit**: Kudelski Security (2017), NCC Group (2019)
- **Known issues**: CVE-2019-6978 (heap-buffer-overflow in hex encoding, does not
  affect crypto primitives). No known backdoors or intentional weaknesses.
- **Independence**: Own implementations (ref10 for Ed25519, donna for X25519)

### BoringSSL
- **Provenance**: Google, fork of OpenSSL
- **Audit**: Continuous internal Google security review
- **Known issues**: Inherits some OpenSSL CVE history but is actively maintained
  and diverged from OpenSSL. Pin to post-fix commit for any relevant CVE.
- **Independence**: Shares some historical code with OpenSSL but is independently
  maintained. libsignal delegates to BoringSSL (via aws-lc-rs/ring) for
  X25519/Ed25519 — NOT independent from libsignal for those primitives.

### Monocypher
- **Provenance**: Loup Vaillant, designed for auditability (~2000 lines)
- **Audit**: Community-audited, compact enough for line-by-line review
- **Known issues**: None known. No Dual_EC or RNG concerns (no RNG in library).
- **Independence**: Completely independent — no shared code with any other oracle

### libsignal
- **Provenance**: Signal Foundation
- **Audit**: NCC Group (2023)
- **Known issues**: None affecting protocol logic. Low-level crypto delegates to
  ring/aws-lc-rs (BoringSSL-derived).
- **Scope**: Protocol oracle ONLY (X3DH, PQXDH, Double Ratchet). NOT used as
  primitive oracle — would create circular trust with BoringSSL.

### PQClean
- **Provenance**: NIST Post-Quantum Cryptography project clean-room implementations
- **Audit**: Academic review via NIST PQC competition
- **Known issues**: None known for ML-KEM-768 (Kyber)
- **Independence**: Independent from liboqs (liboqs includes PQClean internally)

### liboqs
- **Provenance**: Open Quantum Safe project
- **Audit**: Academic/community review
- **Known issues**: Build with -DOQS_USE_OPENSSL=OFF to avoid pulling OpenSSL
- **Independence**: Includes PQClean algorithms internally — NOT fully independent
  from PQClean for shared algorithms. Use as a SECOND PQ oracle, not as the
  primary if PQClean is already primary.

### RustCrypto
- **Provenance**: RustCrypto project (community crates on crates.io)
- **Audit**: No formal audit. Widely used in Rust ecosystem.
- **Known issues**: Various crate-specific CVEs — pin to post-fix versions.
- **Independence**: Independent implementations per crate (sha2, aes-gcm, etc.)

## RNG Concerns

- **Dual_EC_DRBG**: No oracle in our suite uses or has ever used Dual_EC_DRBG.
- **Oracle PRNG**: For differential testing, all random inputs are generated by
  a ChaCha20-based DRBG with a recorded 32-byte seed. Oracles do not generate
  their own randomness — they process deterministic inputs.
